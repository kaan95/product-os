# Tone and Calibration

**Be concrete, not corporate.** Roadmaps that say "drive engagement" or "improve the customer experience" signal that the team hasn't done the hard thinking. Push for specifics. "Daily active users increase from 12K to 18K" is better than "grow engagement."

**Be honest about uncertainty.** If the team doesn't have a baseline for a KR, say so and suggest how they could establish one. A KR like "retention improves (TBD baseline)" is more credible than a made-up number.

**Match the user's level of detail.** If they give you a sketch, don't produce a 10-page plan. If they give you a rich set of notes, produce a detailed roadmap. The goal is the minimum structure that serves them — not a comprehensive template filled with Lorem ipsum.

**Preserve what's good.** If the user has already done outcome thinking (even partially), honour it and build on it rather than replacing it wholesale.

---

# Edge Cases

**"I have too many objectives."**
More than 3–4 objectives in a quarter usually means the team is trying to do everything. Gently flag this. Suggest clustering related objectives, or splitting into a primary focus and a secondary track. A team that can't say what they're NOT doing has no real strategy.

**"Our stakeholders just give us features to build."**
This is common. Help the user translate stakeholder feature requests into the underlying outcomes the stakeholder actually cares about, then frame the roadmap around those outcomes. The features become initiatives (bets) under the relevant KR.

**"We don't have baseline metrics."**
That's fine for now — mark the baseline as "TBD" and include a first-week action to establish it. A roadmap that honestly says "we don't know our baseline yet" is more trustworthy than one that invents numbers.

**"Some of our work is hard to tie to any outcome."**
It belongs in Committed Work, or it's a signal that the work shouldn't be on the roadmap at all. Ask the user: "If this work gets delayed a month, would anyone notice or care? What would get worse?" If the answer is nothing, it might not be worth the quarter.

# Output Instructions

Save the roadmap to the user's workspace folder (e.g. `/sessions/.../mnt/Product Management OS/`) as `Q[N]-[year]-outcome-roadmap.md` (or a more descriptive name if the user specifies one). Provide a direct link to the file.

If you made significant interpretation calls (e.g., you inferred the objective from a list of features, or you reframed a KR), briefly explain your reasoning so the user knows what to review and correct.
