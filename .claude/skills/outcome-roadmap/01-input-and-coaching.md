# Step 1: Read and Assess the Input

Accept any form of input without requiring the user to clean it up first:

- Bullet lists of initiatives or features
- Meeting notes (from Notion, markdown, pasted text)
- Existing OKR or strategy docs (even if output-focused)
- Free-form thoughts or spoken-style notes
- Any combination of the above

**Read everything before you begin.** Don't start restructuring after the first bullet point. Get the full picture — what outcomes the user cares about (even if not stated explicitly), what constraints exist (deadlines, tech debt, committed initiatives), and what's genuinely flexible.

After reading, make a quick mental inventory:
- What items are **true outcomes** (measurable changes in behaviour, retention, revenue, satisfaction)?
- What items are **outputs** (features, projects, releases) that might *contribute to* an outcome but aren't outcomes themselves?
- What items are **constraints or commitments** that have to happen regardless (e.g., compliance work, a pre-existing contract)?
- What's **missing** — is there a clear objective, or just a wish list?

---

# Step 2: Decide Whether to Coach or Structure (or Both)

**If the input is mostly outputs** (e.g. "redesign onboarding", "ship push notifications", "migrate to new infra"):
→ Go into coaching mode. Ask the user what *outcome* each of these is in service of. Don't just relabel — push them to articulate the actual behavioural change they're hoping for.

**If the input contains a mix** (some outcomes stated, some implied, some pure outputs):
→ Structure what's clear, and flag what needs coaching. Offer a "best attempt" framing for the unclear ones and ask the user to validate.

**If the input is already outcome-oriented**:
→ Go straight to structuring. Validate the OKR hierarchy and produce the roadmap.

**Adaptive rule:** When in doubt, make a good-faith attempt to frame outcomes from the input, then explicitly show your reasoning and ask the user to confirm or correct. Don't block on perfect input. A near-miss outcome framing that the user can react to is more useful than a long questionnaire.
